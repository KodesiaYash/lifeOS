## 1. Schema Conventions

### 1.1 Naming

- **Tables**: `{namespace}_{entity}` in snake_case plural (e.g., `core_tenants`, `health_meals`)
- **Primary keys**: `id` — UUID v4, generated by the application
- **Foreign keys**: `{referenced_table_singular}_id` (e.g., `tenant_id`, `user_id`, `conversation_id`)
- **Timestamps**: `created_at`, `updated_at`, `deleted_at` — all `TIMESTAMPTZ`, all UTC
- **JSONB columns**: Named `metadata`, `config`, `settings`, or `payload` depending on purpose
- **Boolean columns**: Prefixed with `is_` or named as adjectives (e.g., `active`, `is_deterministic`)

### 1.2 Standard Columns

Every user-data table includes:

```sql
id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
tenant_id       UUID NOT NULL REFERENCES core_tenants(id),
created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
deleted_at      TIMESTAMPTZ          -- NULL = not deleted (soft delete)
```

### 1.3 Types Used

```sql
-- Custom types
CREATE TYPE channel_type AS ENUM ('whatsapp', 'telegram', 'rest_api', 'web_app', 'email', 'slack', 'voice');
CREATE TYPE message_direction AS ENUM ('inbound', 'outbound');
CREATE TYPE content_type AS ENUM ('text', 'image', 'audio', 'video', 'document', 'location', 'contact', 'interactive', 'system');
CREATE TYPE workflow_status AS ENUM ('pending', 'running', 'waiting_input', 'completed', 'failed', 'cancelled');
CREATE TYPE job_status AS ENUM ('pending', 'running', 'completed', 'failed', 'skipped');
CREATE TYPE task_status AS ENUM ('pending', 'in_progress', 'completed', 'cancelled');
CREATE TYPE task_priority AS ENUM ('low', 'medium', 'high', 'urgent');
CREATE TYPE schedule_type AS ENUM ('once', 'recurring', 'cron', 'spaced_repetition');
CREATE TYPE tenant_role AS ENUM ('owner', 'admin', 'member', 'viewer');
CREATE TYPE workspace_type AS ENUM ('personal', 'shared', 'team');
```

### 1.4 Row-Level Security Template

```sql
-- Applied to every user-data table
ALTER TABLE {table} ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON {table}
    USING (tenant_id = current_setting('app.current_tenant_id')::uuid);

CREATE POLICY tenant_isolation_insert ON {table}
    FOR INSERT
    WITH CHECK (tenant_id = current_setting('app.current_tenant_id')::uuid);
```

## 2. Core Entities

### 2.1 core_tenants

```sql
CREATE TABLE core_tenants (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(255) NOT NULL,
    slug            VARCHAR(63) NOT NULL UNIQUE,   -- URL-safe identifier
    plan            VARCHAR(50) NOT NULL DEFAULT 'personal',  -- 'personal', 'family', 'pro', 'enterprise'
    settings        JSONB NOT NULL DEFAULT '{}',   -- Tenant-wide settings
    active          BOOLEAN NOT NULL DEFAULT true,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- No tenant_id here — this IS the tenant table.
-- No soft delete — tenant deactivation uses the 'active' flag.
```

### 2.2 core_users

```sql
CREATE TABLE core_users (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email           VARCHAR(255) UNIQUE,           -- May be NULL for channel-only users
    name            VARCHAR(255),
    display_name    VARCHAR(100),
    auth_provider   VARCHAR(50),                   -- 'local', 'google', 'github', etc.
    auth_provider_id VARCHAR(255),                 -- Provider-specific user ID
    password_hash   VARCHAR(255),                  -- Only for 'local' auth
    avatar_url      VARCHAR(1024),
    timezone        VARCHAR(50) DEFAULT 'UTC',
    language        VARCHAR(10) DEFAULT 'en',

    last_login_at   TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()

-- No tenant_id — users exist globally and are linked to tenants via core_tenant_users.
```

### 3.5 comm_messages

```sql
CREATE TABLE comm_messages (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id           UUID NOT NULL REFERENCES core_tenants(id),
    conversation_id     UUID NOT NULL REFERENCES comm_conversations(id),
    direction           message_direction NOT NULL,
    content_type        content_type NOT NULL DEFAULT 'text',
    text                TEXT,
    media_ref           VARCHAR(1024),             -- Object storage path or URL
    media_mime_type     VARCHAR(100),
    channel_message_id  VARCHAR(255),              -- Channel-provider's message ID
    idempotency_key     VARCHAR(255) NOT NULL,
    correlation_id      UUID,                      -- Links to triggering event or workflow
    metadata            JSONB NOT NULL DEFAULT '{}',
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),

    UNIQUE (tenant_id, idempotency_key)
) PARTITION BY RANGE (created_at);

-- Create monthly partitions
CREATE TABLE comm_messages_2026_01 PARTITION OF comm_messages
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
CREATE TABLE comm_messages_2026_02 PARTITION OF comm_messages
    FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');
-- ... continue for each month

CREATE INDEX idx_messages_conversation ON comm_messages(conversation_id, created_at DESC);
CREATE INDEX idx_messages_correlation ON comm_messages(correlation_id);
CREATE INDEX idx_messages_idempotency ON comm_messages(tenant_id, idempotency_key);
```

## 4. Memory Entities

### 4.1 mem_facts

Long-term structured memory facts — discrete, retrievable pieces of information about the user.

```sql
CREATE TABLE mem_facts (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID NOT NULL REFERENCES core_tenants(id),
    user_id         UUID NOT NULL REFERENCES core_users(id),
    domain          VARCHAR(100),                  -- Originating domain, NULL for cross-domain
    category        VARCHAR(100) NOT NULL,         -- 'preference', 'goal', 'habit', 'fact', 'setting'
    key             VARCHAR(255) NOT NULL,         -- e.g., "dietary_preference", "wake_up_time"
    value           TEXT NOT NULL,                 -- Human-readable value
    structured_value JSONB,                        -- Machine-readable value
    confidence      FLOAT NOT NULL DEFAULT 1.0,    -- 0-1 how confident we are
    source          VARCHAR(50) NOT NULL,          -- 'user_stated', 'system_inferred', 'connector'
    source_event_id UUID,                          -- Event that created this fact
    active          BOOLEAN NOT NULL DEFAULT true,
    superseded_by   UUID REFERENCES mem_facts(id), -- If this fact was updated/replaced
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at      TIMESTAMPTZ
);

CREATE INDEX idx_facts_tenant_user ON mem_facts(tenant_id, user_id);
CREATE INDEX idx_facts_domain ON mem_facts(tenant_id, user_id, domain);
CREATE INDEX idx_facts_category ON mem_facts(tenant_id, user_id, category);
CREATE INDEX idx_facts_key ON mem_facts(tenant_id, user_id, key);
```

### 4.2 mem_semantic_memories

Long-term semantic memories — summaries, patterns, insights, behavioral observations.

```sql
CREATE TABLE mem_semantic_memories (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID NOT NULL REFERENCES core_tenants(id),
    user_id         UUID NOT NULL REFERENCES core_users(id),
    memory_type     VARCHAR(50) NOT NULL,          -- 'summary', 'pattern', 'insight', 'preference', 'observation'
    content         TEXT NOT NULL,                 -- Natural language content
    embedding       vector(1536),                  -- pgvector embedding
    source_domain   VARCHAR(100),                  -- Originating domain, NULL for cross-domain
    related_domains TEXT[] NOT NULL DEFAULT '{}',
    confidence      FLOAT NOT NULL DEFAULT 1.0,
    importance      FLOAT NOT NULL DEFAULT 0.5,    -- 0-1 importance for retrieval ranking
    access_count    INTEGER NOT NULL DEFAULT 0,
    last_accessed_at TIMESTAMPTZ,
    superseded_by   UUID REFERENCES mem_semantic_memories(id),
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at      TIMESTAMPTZ
);

CREATE INDEX idx_semantic_mem_tenant_user ON mem_semantic_memories(tenant_id, user_id);
CREATE INDEX idx_semantic_mem_type ON mem_semantic_memories(tenant_id, user_id, memory_type);
CREATE INDEX idx_semantic_mem_domain ON mem_semantic_memories(tenant_id, user_id, source_domain);
CREATE INDEX idx_semantic_mem_embedding ON mem_semantic_memories
    USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);
```

## 5. Knowledge Entities

### 5.1 know_documents

Metadata for ingested knowledge documents (articles, PDFs, notes, etc.).

```sql
CREATE TABLE know_documents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID NOT NULL REFERENCES core_tenants(id),
    user_id         UUID NOT NULL REFERENCES core_users(id),
    source_type     VARCHAR(50) NOT NULL,          -- 'url', 'file_upload', 'note', 'browser_capture'
    url             VARCHAR(2048),
    title           VARCHAR(1000),
    author          VARCHAR(500),
    description     TEXT,
    summary         TEXT,                          -- AI-generated summary
    key_points      JSONB,                         -- AI-generated key takeaways
    content_type    VARCHAR(100),                  -- MIME type or content category
    language        VARCHAR(10),
    word_count      INTEGER,
    chunk_count     INTEGER NOT NULL DEFAULT 0,
    content_hash    VARCHAR(64),                   -- SHA-256 for deduplication
    storage_path    VARCHAR(1024),                 -- Object storage path for raw content
    domain_tags     TEXT[] NOT NULL DEFAULT '{}',   -- Which domains this is relevant to
    topic_tags      TEXT[] NOT NULL DEFAULT '{}',   -- Extracted topics
    user_tags       TEXT[] NOT NULL DEFAULT '{}',   -- User-provided tags
    status          VARCHAR(50) NOT NULL DEFAULT 'pending',  -- 'pending', 'processing', 'completed', 'failed'
    error_message   TEXT,
    source_channel  VARCHAR(50),                   -- Channel the content was captured from
    captured_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    processed_at    TIMESTAMPTZ,
    last_accessed_at TIMESTAMPTZ,
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at      TIMESTAMPTZ
);

CREATE INDEX idx_documents_tenant_user ON know_documents(tenant_id, user_id);
CREATE INDEX idx_documents_status ON know_documents(tenant_id, status);
CREATE INDEX idx_documents_domain_tags ON know_documents USING gin(domain_tags);
CREATE INDEX idx_documents_topic_tags ON know_documents USING gin(topic_tags);
CREATE INDEX idx_documents_content_hash ON know_documents(tenant_id, content_hash);
CREATE INDEX idx_documents_url ON know_documents(tenant_id, url);
CREATE INDEX idx_documents_fts ON know_documents USING gin(
    to_tsvector('english', coalesce(title, '') || ' ' || coalesce(summary, ''))
);
```

### 5.2 know_chunks

```sql
CREATE INDEX idx_chunks_document ON know_chunks(document_id, chunk_index);
CREATE INDEX idx_chunks_tenant ON know_chunks(tenant_id, user_id);
CREATE INDEX idx_chunks_embedding ON know_chunks
    USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);
CREATE INDEX idx_chunks_fts ON know_chunks USING gin(to_tsvector('english', text));
```

## 15. Index Strategy

### 15.1 Primary Access Patterns and Indexes

| Access Pattern | Table | Index |
|---------------|-------|-------|
| Load user's recent messages | `comm_messages` | `(conversation_id, created_at DESC)` |
| Find conversation by channel identity | `comm_conversations` | `(channel_identity_id)` |
| Resolve channel identity | `comm_channel_identities` | `(channel_account_id, external_user_id)` |
| Dedup inbound messages | `comm_messages` | `(tenant_id, idempotency_key)` |
| Semantic search (memories) | `mem_semantic_memories` | HNSW on `embedding` |
| Semantic search (knowledge) | `know_chunks` | HNSW on `embedding` |
| Full-text search (knowledge) | `know_chunks` | GIN on `tsvector(text)` |
| Full-text search (documents) | `know_documents` | GIN on `tsvector(title, summary)` |
| Filter by domain tags | `know_documents` | GIN on `domain_tags` |
| Pending scheduled jobs | `sched_job_instances` | Partial on `(scheduled_for) WHERE status = 'pending'` |
| User's active tasks | `sched_tasks` | `(tenant_id, user_id, status)` |
| Events by type | `evt_events` | `(tenant_id, event_type, timestamp DESC)` |
| Events by domain | `evt_events` | `(tenant_id, domain, timestamp DESC)` |
| Trace by correlation | `evt_events` | `(correlation_id)` |
| User's meals by date | `health_meals` | `(tenant_id, user_id, logged_at DESC)` |
| Transactions by date | `fin_transactions` | `(tenant_id, user_id, transaction_date DESC)` |
| Transactions by category | `fin_transactions` | `(tenant_id, user_id, category_id, transaction_date DESC)` |
| Vocabulary due for review | `edu_vocabulary` | Partial on `(tenant_id, user_id, sr_next_review) WHERE deleted_at IS NULL` |

### 15.2 Vector Index Configuration

pgvector HNSW indexes are used for approximate nearest neighbor search:

```sql
-- Configuration parameters:
-- m = 16 (number of connections per node, higher = more accurate but more memory)
-- ef_construction = 64 (higher = better index quality, slower build)
-- Using cosine distance (vector_cosine_ops) for normalized embeddings

CREATE INDEX idx_semantic_mem_embedding ON mem_semantic_memories
    USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);

CREATE INDEX idx_chunks_embedding ON know_chunks
    USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);

CREATE INDEX idx_conv_summaries_embedding ON mem_conversation_summaries
    USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);
```

At query time, set `ef_search` for quality/speed tradeoff:
```sql
SET hnsw.ef_search = 40;  -- Default: 40. Higher = more accurate, slower.
```

## 16. Partitioning Strategy

### 16.1 Partitioned Tables

| Table | Partition Key | Partition Size | Rationale |
|-------|--------------|----------------|-----------|
| `comm_messages` | `created_at` | Monthly | High write volume, time-series access pattern |
| `evt_events` | `timestamp` | Monthly | Append-only, time-series, high volume |
| `audit_agent_invocations` | `timestamp` | Quarterly | Lower volume than events, longer retention |

### 16.2 Partition Management

- **Automated creation**: A scheduled job creates next month's partition 7 days before month end.
- **Retention**: Old partitions can be detached and moved to cold storage (e.g., after 2 years for messages, never for events).
- **Detach command**: `ALTER TABLE comm_messages DETACH PARTITION comm_messages_2024_01;`
